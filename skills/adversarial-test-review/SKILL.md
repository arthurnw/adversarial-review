# Adversarial Test Review

Six-agent process for pressure-testing tests — especially LLM-generated ones.
Three reviewers scan the in-scope tests in parallel through three distinct
smell lenses, then three adversarial challengers pressure-test their findings.
The result is a curated, high-signal list of weak tests, each with a concrete
recommended fix, presented one at a time for your reaction.

The dominant failure mode of this kind of review is the **false positive** —
calling a legitimately simple test "trivial", or recommending deletion of a test
that quietly guards a real regression. The challenger pass exists to cut exactly
that.

## Arguments

- `[target]` — optional. What to review. One of:
  - **omitted** (default): staged test changes; if none are staged, the latest
    commit that touched test files. See §1.
  - a **git ref or range** — e.g. `origin/main`, `HEAD~3..HEAD`, a branch, a SHA.
    Review the test changes in that diff.
  - one or more **paths/globs** to test files — review those files in full,
    regardless of git state. Use when reviewing tests that aren't a recent change.

There is intentionally no migration-cost flag — migration cost is irrelevant to
whether a test earns its place.

## Workflow

### 0. Pick the dispatch adapter

This skill is harness-agnostic. The actual subagent-launch calls live in a sibling
reference file — load the right one now, before proceeding:

- If a `subagent` tool is available (pi with `pi-subagents` installed), read
  [`dispatch-pi.md`](dispatch-pi.md) and follow its dispatch patterns. The preflight
  check in that file runs **before** step 1 — it may tell you to bail out early if
  the extension failed to load.
- Otherwise (Claude Code), read [`dispatch-claude-code.md`](dispatch-claude-code.md)
  and follow its `Task`-tool dispatch patterns.

If the environment is ambiguous, ask the user once before continuing.

### 1. Resolve the test scope

First learn this repo's test conventions — list a few existing test files to see
how they're named and located (e.g. `*_spec.rb` under `spec/`, `*_test.go`,
`test_*.py`/`*_test.py` under `tests/`, `*.test.ts`/`*.spec.ts`, `*.test.tsx`,
`__tests__/`, `*Test.java`). Use what you find to build the path filter below.

Then resolve scope in this order:

```bash
# (a) explicit target argument?  If a ref/range, diff it; if paths, use them directly.

# (b) staged test changes
git diff --cached --name-only        # filter to test files by the conventions above
git diff --cached -- <test files>    # the staged test diff

# (c) else: the latest commit that touched test files
git log -1 --format=%H -- '*_spec.rb' '*_test.*' 'test_*.py' '*.test.*' '*.spec.*' \
  'tests/**' 'spec/**' '__tests__/**'
git show <that-sha> -- <test files>  # the test diff in that commit
```

If nothing resolves to a test file, stop and ask the user what to review. Report
the resolved scope before dispatching: which files, and how it was resolved
(staged / commit `<sha>` / explicit), so the user can redirect if it's wrong.

**Reviewers need more than the test diff.** They must also read the **production
code under test** to judge circularity, library-testing, and missing edge cases.
The reviewer prompt instructs them to do this; make sure the scope you hand off
names the test files clearly enough that a fresh agent can find both the tests and
the code they target.

### 2. Dispatch three reviewer agents in parallel

Use the adapter's parallel-dispatch pattern. Each reviewer gets the resolved scope
(the in-scope test files and how to obtain their content) and one of the three
lens mandates below. Each reviewer reviews **all** in-scope tests; the lens is its
**primary** focus, not a hard boundary — it should flag any smell it sees.

**Reviewer prompt template (assembled by the adapter):**

```
You are adversarially reviewing the quality of tests on a code change. Many were
likely generated by an LLM. Your job is to find tests that do not earn their place.

## Your lens: {LENS}

{LENS_INSTRUCTIONS}

## Tests in scope

{SCOPE_INSTRUCTIONS}
  (e.g. the exact files, and the git command to see their content / diff)

## How to judge

- Read the test files in scope AND the production code they exercise. You cannot
  tell a circular or library-testing test from the test alone.
- For each test, ask: if the application logic broke, would this test fail? If the
  answer is "no" or "not for the right reason", it's suspect.
- Identify the test framework from the files and apply its idioms (fixtures, mocks,
  matchers) — judge against good practice for THAT framework.
- Be concrete and specific. A senior engineer reading your finding should be able
  to act on it without re-deriving it.

## Smell taxonomy (your lens owns the ones marked, but flag any you see)

{TAXONOMY}

## Output format

Return a structured list of findings. For each suspect test:
- ID: Sequential (R1, R2, R3...)
- Test: file:line and the test/example name
- Smell: which taxonomy category
- Severity: High / Medium / Low (how confident + how much it misleads)
- Why it's weak: the actual problem, referencing the code under test
- Recommended action: Delete / Rewrite-to-assert-behavior / Strengthen / Move
  (e.g. to an integration test) / Keep-with-note
- Suggested fix: concrete and specific (what to assert instead, what to stop
  mocking, which edge case to add) — not vague advice

End with a one-line overall assessment of the test suite's health in your lens.
```

The shared **taxonomy** block, substituted into `{TAXONOMY}` for every reviewer:

```
1. Trivial / no-op — asserts nothing meaningful, or asserts a constant
   (`assert True`, `expect(true).toBe(true)`); constructs objects but never asserts.
2. Circular / tautological — asserts the literal value just assigned; re-implements
   the production logic inside the test and asserts they match (a prod bug is
   mirrored in the test); mocks a function to return X then asserts it returned X.
3. Testing the library/framework/stdlib — asserts behavior owned by a dependency
   (the ORM saves a row, the serializer round-trips, the router maps a path) rather
   than this application's own logic.
4. Over-mocked — so much is mocked that the real code path never executes; the test
   only verifies that mocks were called as written.
5. Asserting implementation details — couples to internals (private methods, call
   order, intermediate data structures, log strings) instead of observable
   behavior/output; breaks under refactors that change nothing observable.
6. Snapshot / golden-only — the whole test is a snapshot assertion with no targeted
   check; rubber-stamps whatever output exists today, bugs included.
7. Happy-path-only / missing edge & error cases — only the success path; no nulls,
   empties, boundaries, invalid input, or error/exception handling.
8. Duplicate / redundant coverage — re-tests a path another test already covers;
   parametrized cases that don't add a distinct condition.
9. Flaky / nondeterministic — depends on wall-clock time, timezone, ordering,
   randomness, real network/filesystem, or sleep-based timing.
```

#### Reviewer A — Validity
Lens: does each test actually exercise *this application's* behavior, and can it
fail for the right reason? **Owns** taxonomy 1 (trivial), 2 (circular), 3
(testing-the-library), 4 (over-mocked).
`{LENS_INSTRUCTIONS}`: Focus on tests that can't fail for a real reason — no-ops,
tautologies, tests of dependencies, and tests where mocking has hollowed out the
code path so nothing real runs.

#### Reviewer B — Behavior vs. implementation
Lens: does each test assert observable behavior rather than internals, and does it
earn its place? **Owns** taxonomy 5 (implementation details), 6 (snapshot-only), 8
(duplicate/redundant).
`{LENS_INSTRUCTIONS}`: Focus on brittleness and redundancy — tests coupled to how
the code works rather than what it does, snapshot tests that assert nothing
specific, and tests that re-cover already-covered paths.

#### Reviewer C — Robustness & coverage
Lens: does the test exercise the risky paths, and does it run reliably? **Owns**
taxonomy 7 (happy-path-only / missing edge & error cases), 9 (flaky/nondeterministic).
`{LENS_INSTRUCTIONS}`: Focus on what's NOT tested that should be (error paths,
boundaries, nulls/empties) and on tests that will pass or fail for reasons
unrelated to the code (time, order, randomness, real I/O).

### 3. Dispatch three challenger agents in parallel

Once all three reviewers complete, dispatch three challengers — one per reviewer.
Each gets that reviewer's findings AND the scope. Use the adapter's pattern again.

**Challenger prompt template:**

```
You are an adversarial challenger reviewing another agent's findings about test
quality. The reviewer flagged tests as weak. Your job is twofold:

1. Defend the tests (filter false positives). For each finding, ask: is the
   reviewer right? A test can look trivial yet guard a real regression or pin a
   public contract. An "implementation detail" may BE the contract. A "duplicate"
   may cover a genuinely distinct branch. A "missing edge case" may be deliberately
   covered elsewhere. If the reviewer is wrong or overreaching, say so and remove or
   downgrade the finding. Deleting a good test is worse than keeping a weak one.
2. Expand coverage. Read the tests and the code under test yourself. Did the
   reviewer miss a weak test in their lens? Add it.

Be rigorous, not contrarian. If a finding is solid, uphold it and move on.

## Original reviewer findings

{REVIEWER_OUTPUT}

## Tests in scope

{SCOPE_INSTRUCTIONS}

## Output format

For each original finding, return:
- Original ID: (e.g. R1)
- Verdict: Upheld / Downgraded / Removed
- Reason: why — for Removed/Downgraded, name what the test actually protects
- If downgraded, the new severity

Then list any NEW weak tests the reviewer missed, using the reviewer's finding
format (IDs C1, C2... to distinguish challenger additions), including a concrete
suggested fix.

End with a one-line assessment of the original review's quality.
```

Wait for all three challengers to complete before synthesizing.

### 4. Synthesize

After all challengers complete:

1. **Collect survivors** — findings Upheld or Downgraded (at the new severity),
   plus challenger additions (C-prefixed IDs). Drop anything Removed.
2. **Deduplicate by test** — if multiple reviewers flagged the same test:line,
   merge into one entry listing each smell, and note it was caught by multiple
   lenses (a signal of importance). Keep the strongest recommended action.
3. **Rank** — High severity first, then Medium, then Low. Within a severity, put
   multi-lens findings first.
4. **Group by theme** if natural clusters emerge (e.g. "over-mocked", "happy-path
   only", "snapshot rubber-stamps").

### 5. Present

**First**, show the full synthesized list as a summary table:

```
## Test Review Summary

X weak tests survived adversarial challenge (Y high, Z medium, W low)

| # | Severity | Smell | Test (file:line) | Action | Problem |
|---|----------|-------|------------------|--------|---------|
| 1 | High | Over-mocked | user_service_test.rb:42 | Rewrite | Mocks the repo, asserts only that save was called |
| 2 | Medium | Snapshot-only | Card.test.tsx:18 | Strengthen | toMatchSnapshot with no behavioral assertion |
| ... | ... | ... | ... | ... | ... |
```

**Then**, walk through each finding one at a time:
- Show the full finding: the test, why it's weak (referencing the code under test),
  the recommended action, and the concrete suggested fix.
- Wait for the user's reaction before moving to the next one.
- Track decisions but don't require a formal response — "next", "skip", "agree",
  "fix it", or "disagree" is fine.
- If the user says "fix it" (or similar) on a finding, apply the suggested fix by
  editing the test file, then continue. Default is propose-only; edit only when
  asked.

If the user wants to stop early ("that's enough", "just the high ones"), respect
that and summarize the remaining items briefly.

## Notes

- This skill reviews **test quality**, not whether the tests pass and not the
  production code's correctness. Use the sibling `adversarial-review` for code.
- Reviewers and challengers are independent agents with no shared context. Each
  reads the tests and the code under test fresh. This is intentional — independent
  perspectives catch different things.
- The challenger pass typically removes or downgrades a meaningful fraction of
  initial findings. That is the point: better to surface 6 tests genuinely worth
  fixing than 15 mixed with false positives that would have you delete good tests.
- A clean result (few or no findings) is a valid, useful outcome — say so plainly
  rather than manufacturing nitpicks.
